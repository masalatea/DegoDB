# 2026-06-17 Generated DBAccess SQLite Runtime Adapter First Slice

## Summary

canonical DBAccess PHP output に portable runtime DB adapter を同梱する first slice を追加した。

目的は、Mtool が出力する `dbaccess-*.php` の既存 `$mtooldb` / `connect_mtooldb_if_not_yet()` surface を保ちながら、lightweight SQLite runtime を選べる入口を作ることである。

## Changes

- `mtool/app/project_output_db_access_generator.php`
  - `_support/mtool_runtime_db.php` を生成する。
  - generated base class から support file を `require_once` する。
  - `MTOOL_RUNTIME_DB_DSN=sqlite:/path/app.sqlite` と `MTOOL_RUNTIME_SQLITE_PATH=/path/app.sqlite` を扱う PDO-backed adapter を追加した。
  - MySQL / MariaDB 運用向けに、DSN 未指定時は mysqli-backed adapter を作る。

  - `sample16` の proxy runtime dbclasses autoload を updated generated adapter path に合わせた。
- tests
  - generated DBAccess style class が SQLite DSN で insert / select できる contract test を追加した。

## Runtime Contract

generated DBAccess users can provide either:

```bash
MTOOL_RUNTIME_DB_DSN=sqlite:/absolute/path/app.sqlite
```

or:

```bash
MTOOL_RUNTIME_SQLITE_PATH=/absolute/path/app.sqlite
```

The adapter exposes the legacy-style methods / properties used by generated DBAccess code:

- `query($sql)`
- `execute($sql, $params)`
- `real_escape_string($value)`
- `errno`
- `error`
- result `fetch_row()`

## Generator Boundary

The SQLite runtime adapter should make common DBAccess output portable across MySQL / MariaDB and SQLite. It does not need to absorb every DB-specific behavior into the generator.

Generator-owned scope:

- common CRUD / SELECT output
- scalar prepared parameters
- portable runtime adapter surface
- predictable generated base class contract

Inherited/custom class scope:

- blob / file binding strategy
- vendor-specific SQL
- complex transaction or lock behavior
- advanced JSON / full-text / geospatial / window-function use

This keeps generated output stable while preserving the existing extension model.

## AI-Assisted Boundary Review

Whether a case should be generated by Mtool or implemented in an inherited/custom class can be decided by AI-assisted review before generation.

The review should classify each candidate as:

- `generated`
- `generated_with_options`
- `inherited_custom`
- `manual_runtime`
- `needs_design_review`

The review artifact should include the reason, target method/class, required generator options, inherited/custom implementation notes, and test or smoke expectations. It is advisory by default; it should not automatically apply complex behavior without review.
